Start Here — The Foundation of All Testing
Before a new car reaches the showroom, a trained person sits inside it, drives it, presses every button, checks every light, and confirms everything works as expected. No machine replaces this step completely.

Manual testing is exactly that — a trained tester uses the software by hand, just like a real user would, and checks that everything works correctly. All automation comes after you first understand manual testing well.

What is Manual Testing? Foundation ⌄
👀
Simple Explanation
Imagine you bought a new phone. Before using it daily, you try every app, check the camera, test the buttons, and make sure the charger works. You are doing manual testing! In software, a tester does the same thing — opens the app, tries every feature, and notes down anything that does not work as expected.
Technical Explanation
Manual testing is the process of executing test cases by a human tester without the use of automation tools. The tester interacts with the application under test (AUT) directly — entering inputs, observing outputs, and comparing actual results against expected results defined in test cases. It is most effective for exploratory testing, usability testing, and ad-hoc testing where human judgement is essential.
Manual testing is the starting point for every QA engineer. Even if you move to automation later, you must first understand how to test manually — because automation only does what you tell it to do.
2
Types of Manual Testing — Explained in Detail Core Concepts ⌄
🗂️
Simple Explanation
Testing is not just one activity — it is a collection of different approaches, each designed to catch a specific type of problem. A good QA tester knows which type to apply and when.
⚙️ 1. Functional Testing
Verifies that each feature of the application works exactly as specified in the requirements. It is the most fundamental type of testing — every feature must be tested before anything else.
What to check: Input fields accept correct values, buttons perform the right actions, forms submit data correctly, error messages appear when required.

Example: On a login page — does entering the correct email and password navigate the user to the dashboard? Does clicking "Forgot Password" open the reset flow?

When to use: After every new feature is developed, before release, and after any bug fix.
💨 2. Smoke Testing
A quick, high-level test to check whether the most critical functionality of the application is working. It is done before detailed testing begins — to confirm the build is stable enough to test further.
What to check: Can the app be launched? Can the user log in? Does the home page load? Are the main navigation links working?

Example: A new build is given by the developer. Before running 200 test cases, the tester first checks — does the app open? Does login work? If these fail, the build is rejected immediately without wasting time on further tests.

When to use: Every time a new build is received from the development team.
Smoke testing saves time. If the app cannot even open, there is no point testing anything else.
🎯 3. Sanity Testing
A focused check done after a bug fix or a small change — to confirm that the specific fix works and has not caused new issues in that area. Unlike smoke testing (which is broad), sanity testing is narrow and targeted.
What to check: Only the area that was changed or fixed is tested in detail.

Example: Developer fixes a bug where the "Apply Coupon" button was not working. The tester performs a sanity test — tests only the coupon functionality to confirm it now works correctly, without re-testing the entire checkout flow.

When to use: After a bug fix or a minor change, when there is limited time.
🔁 4. Regression Testing
Re-tests previously working functionality to ensure that new code changes have not broken anything. Every time the codebase changes, regression testing is done to protect existing features.
What to check: All features that were working before the change must still work after the change.

Example: A new "Dark Mode" feature is added to the app. Regression testing ensures that login, search, checkout, and all other existing features still work correctly — even though only Dark Mode was changed.

When to use: After every code change, bug fix, or new feature addition.
Regression testing is the most repeated type of testing in any project — this is why automation is often applied here first.
🔍 5. Exploratory Testing
The tester freely explores the application without predefined test cases — using knowledge, intuition, and curiosity to discover unexpected bugs. This relies entirely on the tester's experience and thinking.
What to check: Anything that feels unusual, unexpected, or poorly handled — edge cases that no one thought to write a test case for.

Example: The tester opens a shopping app and starts trying unusual things — adding 999 items to the cart, typing emojis in the search bar, rapidly clicking the checkout button multiple times, switching tabs mid-payment. These are scenarios developers may not have considered.

When to use: When exploring a new feature for the first time, or when looking for bugs that structured test cases might miss.
📏 6. Boundary Value Testing
Tests the extreme edges of allowed input ranges. Most defects occur at boundaries — not in the middle. This technique ensures the application handles minimum, maximum, and just-outside values correctly.
What to check: Minimum value, maximum value, one below minimum, one above maximum.

Example: A password field requires 8 to 20 characters.
— 7 characters → should be rejected
— 8 characters → should be accepted (minimum boundary)
— 20 characters → should be accepted (maximum boundary)
— 21 characters → should be rejected

When to use: Any field with a defined minimum or maximum — age, price, character count, quantity, date range.
❌ 7. Negative Testing
Tests the application with invalid, unexpected, or incorrect inputs — to verify it handles errors gracefully and does not crash or behave unpredictably. A well-built application should always respond with a clear, helpful error message.
What to check: Invalid data, empty required fields, wrong formats, special characters in unexpected places, SQL injection attempts.

Example:
— Enter letters in a numeric phone field → should show "Please enter a valid number"
— Submit a form with all fields empty → each required field should show a validation error
— Enter a past date in a "Date of Birth" field that requires adults only → should be rejected

When to use: For every input field and every user action — always test what happens when something goes wrong.
Negative testing is often skipped by beginners — but it is where many critical bugs hide. Always test the wrong path, not just the right one.
🖥️ 8. UI Testing
Verifies the visual and interactive elements of the application — layout, fonts, colours, button positions, spacing, and responsiveness across different screen sizes and browsers.
What to check: Alignment, font consistency, button sizes, colour contrast, broken images, overlapping elements, and how the page looks on mobile vs desktop.

Example:
— The Submit button is cut off on smaller phone screens
— A label overlaps the input field on Firefox but looks fine on Chrome
— An error message appears in white text on a white background — invisible to the user

When to use: After UI changes, after responsive design updates, and when testing on multiple browsers or devices.
👤 9. User Acceptance Testing (UAT)
The final stage of testing — done by the actual end users or business stakeholders to confirm the software meets their needs and is ready for release. UAT is not done by the QA team — it is done by the people who will actually use the product.
What to check: Whether the software solves the real problem it was built for, from the perspective of the actual user — not just technical correctness.

Example: A hospital management system is built and tested by the QA team. Before going live, actual hospital staff — doctors, nurses, front desk — are given access to try it in a test environment. They verify that the workflows match how they actually work in real life.

When to use: After all QA testing is complete, just before the official release.
UAT can catch issues that QA missed — not because QA was wrong, but because real users think and behave differently from testers.
⚡ 10. Ad-hoc Testing
Informal testing with no documentation, no test cases, and no plan — the tester randomly tests the application trying to break it. It relies entirely on the tester's intuition and domain knowledge.
What to check: Anything — there are no rules. The tester goes wherever their instinct leads.

Example: A tester opens an e-commerce app and starts randomly — searches for a product with only spaces, adds it to the cart, removes it, adds it again, applies a coupon, removes the coupon, goes back, refreshes — just to see if anything breaks unexpectedly.

When to use: When time is short, when exploring a completely new area of the app, or as a complement to structured testing.

How to Write a Test Case Must Know ⌄
📝
Simple Explanation
A test case is like a recipe. A recipe tells you: what ingredients you need, the steps to follow, and what the dish should look like when done. A test case tells you: what conditions are needed, what steps to perform, and what the correct result should be. Anyone should be able to follow a well-written test case and get the same result.
Technical Explanation
A test case is a documented set of conditions, inputs, steps, and expected outcomes used to validate a specific functionality. A well-written test case includes: Test Case ID, Title, Preconditions, Test Steps, Test Data, Expected Result, Actual Result, Status (Pass/Fail), and Priority (Critical/High/Medium/Low).

Example — Login Feature Test Case:

Test Case Format 
Test Case ID   : TC_LOGIN_001
Title          : Verify successful login with valid credentials
Priority       : Critical

Precondition   : User account must exist. App must be open.

Test Steps     :
  1. Open the application and navigate to the Login page
  2. Enter a valid email address in the Email field
  3. Enter the correct password in the Password field
  4. Click the Login button

Test Data      :
  Email    : testuser@example.com
  Password : Test@1234

Expected Result: User is navigated to the Dashboard page.
                  Welcome message is displayed with the user's name.

Actual Result  : [Fill after executing the test]
Status         : Pass / Fail
Always write test cases before you start testing — not while testing. This keeps your approach structured and ensures nothing is missed.
Must Know ⌄
🐛
Simple Explanation
When you find a problem, you need to report it clearly so the developer can fix it. A good bug report answers three questions: What did you do? What did you expect to happen? What actually happened? The clearer your report, the faster the developer finds and fixes the problem.
Technical Explanation
A defect report (bug report) is a formal document submitted to a bug tracking tool (Jira, Azure DevOps, Bugzilla) that captures a software defect. Severity indicates the impact on the system (Critical, High, Medium, Low). Priority indicates how urgently it must be fixed. Both are determined by the QA engineer and reviewed by the project manager.
Bug Report Format 
Bug ID         : BUG_LOGIN_003
Title          : Login button unresponsive when password field is empty
Severity       : High
Priority       : High
Environment    : Chrome 124, Windows 11, App v2.3.1

Steps to Reproduce:
  1. Open the application Login page
  2. Enter a valid email address
  3. Leave the Password field completely empty
  4. Click the Login button

Expected Result:
  A validation message should appear:
  "Password is required. Please enter your password."

Actual Result  :
  Nothing happens. The Login button does not respond.
  No error message is shown. No page navigation occurs.

Always attach a screenshot or screen recording to your bug report. It removes all confusion and helps the developer reproduce the issue immediately.
Process ⌄
🔄
Simple Explanation
Testing in a real project follows a clear process — just like how a book goes from author → editor → proofreader → printer → bookshop. Each step has a purpose. Skipping a step causes problems later. This process is called the Software Testing Life Cycle (STLC).
  • Requirement Analysis — Read and understand what the feature should do. Ask questions if anything is unclear.
  • Test Planning — Decide what to test, how to test it, who will test it, and how long it will take.
  • Test Case Design — Write all test cases covering happy path, negative, boundary, and edge cases.
  • Test Environment Setup — Prepare the system, test data, and tools needed to run the tests.
  • Test Execution — Run each test case one by one. Mark the result as Pass or Fail.
  • Bug Reporting — Log every failure as a bug in the tracking tool with clear steps and screenshots.
  • Re-testing & Regression — After the developer fixes the bug, test it again. Also check that the fix did not break anything else.
  • Test Closure — Summarise the results, prepare a test summary report, and confirm the feature is ready to release.
Understanding this life cycle is the single most important foundation for any QA career. Master this first — everything else builds on top of it.

Entry Criteria and Exit Criteria — When Testing Starts and Ends Process ⌄
🚦
Simple Explanation
Think of a cricket match. The match only starts when the pitch is ready, both teams have arrived, and the umpires are in place — these are Entry Criteria. The match ends only when all overs are bowled or one team wins — these are Exit Criteria. In testing, we define similar conditions to decide when testing can start and when it is considered complete.
Technical Explanation
Entry Criteria — Conditions that must be met before testing begins. Ensures testers are not wasting time on an unstable build.

Exit Criteria — Conditions that must be met before testing is declared complete. Ensures the product meets the required quality standard before release.
Entry and exit criteria are written in the Test Plan. They protect the team — entry criteria prevent testing too early, exit criteria prevent releasing too soon.

7 Principles of Software Testing Foundation ⌄
📐
Simple Explanation
Just like there are rules in a game, software testing has 7 fundamental principles every QA engineer must know. These principles guide how we think about testing — they help us test smarter, not just more. Knowing these principles is the difference between a beginner tester and a professional QA engineer.
1️⃣
Testing shows presence of bugs
Testing can prove bugs exist — but can never prove the software is 100% bug-free. Even after thorough testing, hidden bugs may remain.
2️⃣
Exhaustive testing is impossible
You cannot test every possible combination of inputs. Focus effort on high-risk and high-impact areas using risk-based testing.
3️⃣
Early testing saves money
A bug found during requirements costs almost nothing to fix. The same bug found after release can cost 100x more. Start testing early.
4️⃣
Defect clustering
Most bugs are found in a small number of modules. 80% of bugs come from 20% of the code. If one area is buggy — dig deeper.
5️⃣
Pesticide paradox
If you run the same tests repeatedly, they stop finding new bugs — like a pesticide that stops working. Keep updating your test cases.
6️⃣
Testing is context-dependent
How you test a banking app differs from testing a game. Testing approach depends on the type of software, risk, and user.
7️⃣
Absence of errors fallacy
A bug-free product is useless if it does not meet user needs. If you built the wrong thing perfectly, it is still a failure. QA must verify the software solves the right problem.
These 7 principles are the foundation of professional software testing. Learn them thoroughly — they will serve you throughout your QA career.

Read This First — What is API Testing?
You open a food delivery app and search for pizza. Within seconds, a list of nearby pizza restaurants appears. You did not go to each restaurant and ask — your app sent a request to a server, the server looked up the data and sent it back, and the app displayed it. That back-and-forth communication between the app and the server happens through an API.

API Testing means testing that communication directly — without opening the app. You send a request and check: did the server respond correctly? Did it return the right data? Did it reject invalid requests? This is one of the most valuable and in-demand QA skills today.
1
What is an API? — The Simplest Explanation Start Here ⌄
🍽️
Simple Explanation — Restaurant Analogy
Think of a restaurant. You (the customer) sit at a table. The kitchen prepares the food. But you do not walk into the kitchen yourself — a waiter takes your order to the kitchen and brings the food back to you.

In software:
— You = the app or browser (client)
— Waiter = the API
— Kitchen = the server and database

The API carries your request to the server and brings the response back. You never see the kitchen — you just get your food (data).
Technical Definition
API stands for Application Programming Interface. It is a set of rules that defines how two software systems communicate with each other. The most common type used in web and mobile apps is a REST API — it uses HTTP (the same protocol your browser uses to load websites) to send and receive data in JSON format.
Technical Explanation
API (Application Programming Interface) is a set of defined rules and protocols that allows two software applications to communicate. In web development, this typically refers to REST APIs which use HTTP as the transport protocol and JSON as the data format.

Key technical terms:
— Endpoint: A specific URL that represents a resource. Example: GET /api/users/1
— Resource: The data entity being accessed (User, Product, Order)
— Client: The application making the request (browser, mobile app, Postman)
— Server: The backend system that processes requests and returns responses
— JSON: JavaScript Object Notation — a lightweight text format for data exchange
— HTTP: Hypertext Transfer Protocol — the communication protocol used by REST APIs

Key Takeaway: "An API is a contract between a client and a server that defines how requests should be made and what responses to expect. REST APIs are stateless — each request contains all the information needed to process it, and the server does not store session state between requests."
As a QA engineer, your job is to test the API directly — not through the app. This lets you verify that the server is working correctly, independent of whether the app UI has any bugs.

HTTP Methods — The Types of Requests You Can Make Core Concept ⌄
📬
Simple Explanation
Think of HTTP methods like actions you can do with a filing cabinet:

— GET = Open the cabinet and read a file (just reading, no changes)
— POST = Add a brand new file into the cabinet
— PUT = Replace an existing file completely
— PATCH = Update only one page inside an existing file
— DELETE = Remove a file from the cabinet permanently
Technical Explanation
HTTP methods define the intended action on a resource. They are also called HTTP verbs.

— GET: Retrieves a resource. Idempotent (same result every call). No body sent. Cached by browsers. Example: GET /api/users returns a list of all users.
— POST: Creates a new resource. Not idempotent (calling twice creates two records). Body contains the new data. Returns 201 Created.
— PUT: Replaces an entire resource. Idempotent. All fields must be provided — missing fields are set to null. Returns 200 OK.
— PATCH: Partially updates a resource. Only the provided fields are changed. Returns 200 OK.
— DELETE: Removes a resource. Idempotent. Returns 204 No Content.

Idempotent means: calling the same request multiple times produces the same result. GET, PUT, DELETE are idempotent. POST is NOT.

Key Takeaway: "GET is read-only and safe. POST creates and is not idempotent. PUT replaces completely and is idempotent. PATCH partially updates. DELETE removes. I test each method by verifying the correct status code, response body, and that the database state changes (or does not) as expected."
As a QA engineer, you test ALL five methods. Make sure GET returns correct data, POST creates correctly, PUT/PATCH updates correctly, and DELETE actually removes the record and returns an error if you try to GET it again.

HTTP Status Codes — What the Server is Telling You Must Know ⌄
🚦
Simple Explanation
Every time you send an API request, the server replies with a 3-digit number called a status code. This number tells you immediately whether things went well or not — and why.

Think of it like a doctor's report:
— 2xx = Everything is fine ✅
— 3xx = Go look somewhere else (redirect)
— 4xx = You made a mistake ❌
— 5xx = The server made a mistake 💥
Technical Explanation
HTTP status codes are standardised 3-digit integers returned by the server to indicate the result of the request. They are grouped into 5 classes:

1xx — Informational: Request received, processing continues. Rare in API testing.
2xx — Success: The request was successfully received, understood, and accepted.
  • 200 OK — Standard success for GET, PUT, PATCH
  • 201 Created — New resource created, usually returned with POST
  • 204 No Content — Success but no body returned, used by DELETE

3xx — Redirection: Client must take additional action (follow a redirect URL).
  • 301 Moved Permanently, 302 Found

4xx — Client Errors: The request contains bad syntax or cannot be fulfilled.
  • 400 Bad Request — Malformed syntax or invalid parameters
  • 401 Unauthorized — Authentication required (no token or invalid token)
  • 403 Forbidden — Authenticated but not authorised for this action
  • 404 Not Found — Resource does not exist
  • 405 Method Not Allowed — HTTP method not supported on this endpoint
  • 422 Unprocessable Entity — Request well-formed but validation failed
  • 429 Too Many Requests — Rate limit exceeded

5xx — Server Errors: The server failed to fulfil a valid request.
  • 500 Internal Server Error — Generic server crash
  • 502 Bad Gateway — Upstream server returned invalid response
  • 503 Service Unavailable — Server temporarily down

Key Takeaway: "In API testing I always verify that the status code matches the operation — 201 for resource creation, 204 for deletion, 400/422 for validation failures, 401 for missing auth, 403 for permission issues, and 404 for non-existent resources. A 200 on an error response or a 500 on bad input are both defects I would raise."
The most important status codes for QA are: 200 (success), 201 (created), 400 (bad input), 401 (not logged in), 404 (not found), and 500 (server crash). Memorise these — they appear in every API test.

Request and Response Structure — What Gets Sent and Received Core Concept ⌄
✉️
Simple Explanation
An API request is like sending a letter. Your letter has:
— An address (the URL — where you are sending it)
— A type of action (the method — GET, POST, etc.)
— Extra instructions (headers — like "I speak JSON" or "here is my login token")
— The actual content (body — the data you are sending)

The server reads your letter and sends one back with:
— A status code (did it work?)
— The response body (the data you asked for)
Technical Explanation
An HTTP request consists of four parts:

1. Request Line: Method + URL + HTTP version. Example: POST /api/users HTTP/1.1
2. Headers: Key-value pairs providing metadata about the request.
  • Content-Type: application/json — tells server the body is JSON
  • Authorization: Bearer <token> — authentication token (JWT)
  • Accept: application/json — tells server what format you want back
  • X-API-Key: abc123 — API key authentication
3. Body: The data payload (only for POST, PUT, PATCH — not GET or DELETE)
4. Query Parameters: Appended to URL for filtering. Example: /api/users?page=1&limit=10

An HTTP response consists of:
1. Status Line: HTTP version + status code + reason phrase
2. Response Headers: Metadata about the response
  • Content-Type: application/json — format of the response body
  • X-RateLimit-Remaining: 99 — how many requests left
3. Response Body: The actual data (usually JSON)

JWT (JSON Web Token): The most common authentication mechanism. It is a Base64-encoded string with three parts: Header.Payload.Signature. The server validates the signature to authenticate requests.

Key Takeaway: "I verify the request structure by checking the Content-Type header, Authorization token, and body schema. In the response I check the status code, Content-Type, all required fields, field data types, and that no sensitive data like passwords or internal IDs are exposed."
JSON (JavaScript Object Notation) is the most common format for API data. It uses curly braces {} for objects and square brackets [] for arrays. It is easy to read — even for non-developers.

API Testing with Postman — Your First API Test Hands-On ⌄
📮
Simple Explanation
Postman is the most popular tool for testing APIs. It is like a remote control for APIs — you type in the URL, choose the method (GET, POST etc.), add any required data, and click Send. The server responds and Postman shows you exactly what came back — the status code, the headers, and the body data.

You do not need to write any code to use Postman. It is a visual tool that anyone can learn in an hour.
  • Download Postman — Go to postman.com and download for free. No account needed to start.
  • Create a new request — Click the + button to open a new tab.
  • Choose the method — Click the dropdown (defaults to GET) and select GET, POST, etc.
  • Enter the URL — Type the API endpoint URL in the address bar.
  • Add headers or body — Click the Headers or Body tab to add data if needed.
  • Click Send — The response appears in the bottom panel with the status code and body.
Technical Explanation
Postman is a GUI-based API client that supports REST, GraphQL, SOAP, and WebSocket testing. Key features for QA:

— Collections: Organise related API requests into folders (like a test suite)
— Environments: Store variables for different environments (Dev, QA, Prod). Example: {{baseUrl}} resolves to https://qa.myapp.com in QA environment

API Test Cases — What to Check in Every API Test Cases ⌄
📋
Simple Explanation
When testing an API you need to check more than just "does it return data?" You need to verify the right data, the right format, the right status code, security, performance, and error handling. Here is a complete checklist of what to test for every API endpoint.
Test TypeExample Test CaseExpected Result
PositiveGET /users/1 with valid token200 OK + correct user data
NegativeGET /users/1 without any token401 Unauthorized
NegativeGET /users/9999 (does not exist)404 Not Found
NegativePOST /users with missing email field400 or 422 with error message
Edge CasePOST /users with email = 10,000 characters400 or 422 — should not crash
SecurityGET /users/2 while logged in as user 1403 Forbidden — cannot see other user data
Technical Explanation
A comprehensive API test strategy covers six dimensions:

1. Functional Testing: Verify the API performs its intended function correctly. Validate request/response schema, business logic, and data accuracy.
2. Schema Validation: Confirm response structure matches the API contract (OpenAPI/Swagger specification). Check field names, data types, required vs optional fields, and nested objects.
3. Authentication and Authorisation Testing:
  • No token → 401 Unauthorized
  • Expired token → 401 Unauthorized
  • Valid token, wrong role → 403 Forbidden
  • IDOR: User A accessing User B's data → 403 Forbidden
4. Error Handling: Verify meaningful error messages (not stack traces), correct error codes, and no sensitive data in error responses.
5. Performance Testing: Response time under normal load (<200ms ideal), behaviour under high concurrent requests (load testing).
6. Security Testing: SQL injection in parameters, XSS in string fields, mass assignment vulnerabilities, rate limiting enforcement.

Boundary value testing for APIs:
  • String fields: empty string, null, max length, special chars, HTML tags, SQL injection
  • Numeric fields: 0, -1, very large number, decimal where integer expected
  • Required fields: omit each required field individually

Key Takeaway: "My API test strategy covers functional correctness, schema validation against the Swagger spec, auth and authorisation (including IDOR), error handling quality, response time benchmarks, and security checks like injection and rate limiting."
Always test the unhappy path (errors, invalid data, missing fields) just as thoroughly as the happy path (valid requests). Most production bugs hide in the error scenarios.

Common API Bugs — What to Look For Bug Hunting ⌄
🐛
Simple Explanation
Experienced API testers know where bugs like to hide. They do not just test the obvious happy path — they probe the edges, send unexpected data, and check for security gaps. Here are the most common API bugs found in real projects.
BugHow to Find ItExpected Behaviour
Wrong status codeCheck every response code against the spec201 for create, 204 for delete, 404 for missing
Missing validationSend empty fields, null values, very long strings400 or 422 with clear error message
No auth checkSend request with no token401 Unauthorized
Data leakageRead all response fields carefullyNo passwords, tokens, or internal IDs exposed
IDORLogin as user 1, access /users/2403 Forbidden
Server error on bad inputSend a number where string expected400, not 500
Technical Explanation
IDOR (Insecure Direct Object Reference): Occurs when an API exposes internal implementation objects (like database IDs) directly in URLs without proper authorisation checks. Example: GET /api/users/2 when logged in as user 1 should return 403, not user 2's data. This is in OWASP Top 10 security vulnerabilities.

Mass Assignment Vulnerability: Occurs when an API blindly accepts all JSON fields in the request body and maps them to database fields. A user could send {"role": "admin"} along with their registration request and gain admin privileges.

SQL Injection via API: Sending SQL code in parameters: GET /api/users?name=' OR 1=1--. A vulnerable API would execute this against the database. Test by sending common injection strings and verifying the API returns a 400, not database data.

Broken Object Level Authorisation (BOLA): OWASP API Security Top 10 #1. API does not validate that the requesting user is authorised to access the specific object they are requesting.

Rate Limiting: APIs should limit requests per IP/user per time window. Test by sending 100+ rapid requests and verifying the API eventually returns 429 Too Many Requests.

Verbose Error Messages: A 500 error that includes a stack trace, database query, or file path is a security vulnerability. Error responses should be generic and not reveal internal implementation details.

Key Takeaway: "Key API security bugs I test for include IDOR (accessing other users' data), broken authentication (bypassing auth), mass assignment (sending extra fields), SQL injection in parameters, missing rate limiting, and verbose error messages exposing internals. I follow the OWASP API Security Top 10 as a testing checklist."
IDOR (Insecure Direct Object Reference) is one of the most common and dangerous security bugs. Always test whether a logged-in user can access another user's data by changing the ID in the URL.

Types of APIs — REST, SOAP, and GraphQL Overview ⌄
🌐
Simple Explanation
APIs come in different styles — like how you can order food by phone, by app, or by walking in. Each style has its own rules. The most common ones you will encounter as a QA engineer are REST, SOAP, and GraphQL.
RESTSOAPGraphQL
Data formatJSON (mostly)XML onlyJSON
Ease of useEasy — widely understoodComplex — lots of XMLMedium
Used byMost modern appsBanks, government, old enterpriseFacebook, GitHub, Shopify
FlexibilityFixed endpoints per resourceFixed contractsAsk only for fields you need

Most common?Yes — learn this firstLess common in new projectsGrowing rapidly
Technical Explanation
REST (Representational State Transfer):
  • Architectural style (not a protocol) — uses HTTP methods and URLs to represent resources
  • Stateless: Each request is independent — server stores no session state
  • Uniform Interface: Resources identified by URLs, manipulated via standard HTTP methods
  • Data format: JSON (primarily), also XML, plain text
  • Test with: Postman, Playwright, RestAssured (Java), Requests (Python)

SOAP (Simple Object Access Protocol):
  • A protocol (stricter rules than REST) — uses XML exclusively
  • Has a WSDL (Web Services Description Language) file that defines all available operations
  • Built-in error handling via SOAP Fault messages
  • Common in banking, healthcare, and enterprise systems
  • Test with: SoapUI, Postman (supports SOAP), ReadyAPI

GraphQL:
  • Query language for APIs — client specifies exactly which fields it needs
  • Single endpoint (e.g., POST /graphql) for all operations
  • Three operation types: Query (read), Mutation (write), Subscription (real-time)
  • Eliminates over-fetching (getting too much data) and under-fetching (too little)
  • Test with: GraphQL Playground, Postman, Playwright

WebSockets: For real-time bidirectional communication (live chat, stock prices). Unlike REST, the connection stays open. Playwright supports WebSocket testing via page.on("websocket").

Key Takeaway: "REST is the most common — stateless, uses HTTP methods, JSON format. SOAP is protocol-based with strict XML contracts via WSDL, found in enterprise/banking. GraphQL uses a single endpoint where clients request only the fields they need, eliminating over-fetching. I have tested REST APIs extensively and have experience with GraphQL. For SOAP I would use SoapUI."
Focus on REST API testing first — it covers 90% of what you will encounter in most QA jobs today. Once comfortable with REST, GraphQL is the next most valuable skill to add.

Writing Your First Test — Login Page Practice ⌄
🏫
Simple Explanation
Think of a security checkpoint. It checks — is this person authorised to enter? A login test does the same — it verifies the right user gets in with valid credentials, and anyone with wrong credentials is blocked with a clear error message.
Example — What We Are Testing
A banking app has a login screen. We want to make sure:
✅ Correct email + password → goes to the account home page
❌ Wrong password entered → shows "Incorrect credentials"
❌ Empty password field → shows "Please enter your password"
Technical Explanation
test.describe() groups related tests together. page.fill() types text into an input field. page.click() presses a button. expect(page).toHaveURL() verifies that the browser navigated to the correct page. Each test runs in a completely fresh browser tab — no shared data between tests.
TypeScript 
// File: tests/login.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Bank App — Login Page', () => {

  // ✅ Test 1: Correct login — should go to home page
  test('valid user can log in successfully', async ({ page }) => {
    await page.goto('https://mybank.com/login');

    // Type the mobile number
    await page.fill('#mobile', '9876543210');

    // Type the PIN
    await page.fill('#pin', '1234');

    // Click the Login button
    await page.click('button[type="submit"]');

    // Verify: Did we reach the home page?
    await expect(page).toHaveURL('/home');
    await expect(page.locator('h1')).toContainText('Welcome');
  });

  // ❌ Test 2: Wrong PIN — should show an error message
  test('wrong PIN shows error message', async ({ page }) => {
    await page.goto('https://mybank.com/login');
    await page.fill('#mobile', '9876543210');
    await page.fill('#pin', '9999');  // wrong PIN
    await page.click('button[type="submit"]');

    // The error message must appear on screen
    await expect(page.locator('.error-message')).toBeVisible();
    await expect(page.locator('.error-message'))
      .toContainText('Incorrect PIN');
  });

});
Notice we tested both the good path (correct login) AND the bad path (wrong PIN). Always test both — that is what makes a thorough QA tester!

How Playwright Works — Step by Step with Diagram Concept ⌄
🔄
Simple Explanation
When you run a Playwright test, a lot happens behind the scenes in a specific order. Your code sends instructions → Playwright talks to the browser → the browser performs the action → Playwright checks the result → it reports Pass or Fail. Understanding this flow helps you write better tests and debug failures faster.
What Happens at Each Step
Step 1 — Your Test Code: You write instructions in TypeScript. Example: "go to login page, fill email, fill password, click submit, check dashboard appears."

Step 2 — Playwright Test Runner: Playwright reads your instructions and translates them into browser commands using the Chrome DevTools Protocol (CDP) or WebKit/Firefox equivalents.

Step 3 — Browser: A real browser (Chrome, Firefox, or Safari) opens and performs the actions — just like a human would, but much faster.

Step 4 — Website: The website responds to each action — form submissions, page navigations, API calls happen in real time.

Step 5 — Report: Playwright compares the result with what you expected. If they match → ✅ Pass. If not → ❌ Fail with a detailed error message and screenshot.
Playwright communicates with browsers using the same protocol that browser DevTools uses — so it can see everything a developer sees.

Variables and Basic Types — Storing Information Safely Basics ⌄
📦
Simple Explanation
A variable is like a labelled box where you store information. In TypeScript, each box has a type label that says what kind of thing it can hold. A box labelled "string" only holds text. A box labelled "number" only holds numbers. If you try to put the wrong thing inside, TypeScript warns you immediately.

This is extremely useful in QA — you never accidentally pass a test name where a test ID number is expected.
TypeScript — QA Variables 
// ── DECLARING VARIABLES WITH TYPES ──────────────────

// string  = text (always in quotes)
let testName: string  = 'Login Test';
let browserName: string = 'Chrome';
let testerName: string  = 'Karthic';

// number  = any number (whole or decimal)
let testCaseId: number  = 1001;

let duration: number    = 1.5;   // seconds

// boolean = true or false ONLY
let isPassed: boolean    = true;
let isBlocked: boolean   = false;
let isRegression: boolean = true;

// ── TYPE SAFETY IN ACTION ────────────────────────────
// TypeScript catches this mistake BEFORE running:
testName = 123;   // ❌ Error: number is not assignable to type string
testCaseId = 'TC-001'; // ❌ Error: string is not assignable to type number

const baseUrl: string = 'https://myapp.com';
const timeout: number = 30000; // 30 seconds

// let = value CAN change (use for test results)

TypeScript — Arrays in QA

const testCaseIds: number[] = [101, 102, 103, 104, 105];

Async / Await — How Playwright Tests Work Must Know ⌄
⏳
Simple Explanation
When you send a WhatsApp message, your phone does not freeze and wait until the message is delivered before you can do anything else. It sends the message in the background and continues. This is asynchronous behaviour.

Playwright tests are asynchronous because browsers take time to respond — a page might take 2 seconds to load. await tells your test: "wait here until this browser action is complete before moving to the next step." Without await, your test would run all steps simultaneously and fail unpredictably.
TypeScript — Async/Await in Playwright 
// Every Playwright test function must be async
// Every browser action must be awaited

import { test, expect } from '@playwright/test';

// ── BASIC ASYNC TEST ─────────────────────────────────
test('User can log in successfully', async ({ page }) => {

  // await = wait for this to finish before next step
  await page.goto('https://myapp.com/login');
  // ↑ waits for page to fully load

  await page.fill('#email', 'test@example.com');
  // ↑ waits for email field to accept text

  await page.fill('#password', 'Test@1234');
  // ↑ waits for password field to accept text

  await page.click('button[type="submit"]');
  // ↑ waits for button click to register

  await expect(page).toHaveURL('/dashboard');
  // ↑ waits for navigation to complete, then checks URL
});

// ── ASYNC HELPER FUNCTION ────────────────────────────
// async function = a function that contains await steps
async function loginUser(page: Page, email: string, password: string): Promise<void> {
  await page.goto('/login');
  await page.fill('#email', email);
  await page.fill('#password', password);
  await page.click('button[type="submit"]');
}

// Using the helper in multiple tests
test('Dashboard loads after login', async ({ page }) => {
  await loginUser(page, 'test@example.com', 'Test@1234');
  await expect(page.locator('h1')).toContainText('Welcome');
});
The most common beginner mistake in Playwright is forgetting await. If you forget it on a step, the test moves to the next line immediately — before the browser finishes the action — and the test fails for no obvious reason. Always add await before every browser action.

Classes — Building Page Object Models OOP ⌄
🏗️
Simple Explanation
A class is a template. Think of it like a cookie cutter — the cutter defines the shape, and every cookie made with it has the same shape but can have different fillings. In Playwright, we use classes to represent each page of the application (Login page, Dashboard page, Checkout page). Each class holds all the selectors and actions for that page in one organised place.
TypeScript — Page Object Model with Classes 
// ── BASE PAGE (shared actions for all pages) ─────────
class BasePage {
  protected page: Page;

  constructor(page: Page) {
    this.page = page;
  }

  async waitForPageLoad(): Promise<void> {
    await this.page.waitForLoadState('networkidle');
  }

  async getTitle(): Promise<string> {
    return await this.page.title();
  }
}

// ── LOGIN PAGE (extends BasePage) ────────────────────
class LoginPage extends BasePage {
  // Selectors — stored as class properties
  private emailInput    = '#email';
  private passwordInput = '#password';
  private loginButton   = 'button[type="submit"]';
  private errorMessage  = '.error-msg';

  // Go to login page
  async goto(): Promise<void> {
    await this.page.goto('/login');
  }

  // Login action — combines all steps
  async login(email: string, password: string): Promise<void> {
    await this.page.fill(this.emailInput, email);
    await this.page.fill(this.passwordInput, password);
    await this.page.click(this.loginButton);
  }

  // Get error message text
  async getError(): Promise<string> {
    return await this.page.locator(this.errorMessage).textContent() ?? '';
  }
}

// ── USING IN A TEST ──────────────────────────────────
test('Valid login redirects to dashboard', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('test@example.com', 'Test@1234');
  await expect(page).toHaveURL('/dashboard');
});

test('Wrong password shows error message', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('test@example.com', 'WrongPass');
  const error = await loginPage.getError();
  await expect(error).toContain('Invalid credentials');
});
Page Object Model (POM) using classes is the industry standard for organising Playwright tests. When a selector changes in the app, you update it in ONE place — the class file — and all tests that use it are automatically fixed.

Error Handling — Writing Reliable Tests Reliability ⌄
🛡️
Simple Explanation
When you drive a car, you wear a seatbelt — not because you plan to crash, but just in case. Error handling in TypeScript works the same way. You wrap your risky code in a try block. If anything goes wrong, the catch block handles it gracefully — instead of crashing the entire test run and giving you no useful information about what failed.
TypeScript — Error Handling in Tests 
// ── BASIC TRY-CATCH ──────────────────────────────────
try {
  // code that might fail goes here
  await page.goto('https://myapp.com/dashboard');
  await expect(page.locator('h1')).toBeVisible();
} catch (error) {
  // runs if anything above fails
  console.error('Test failed:', (error as Error).message);
  await page.screenshot({ path: 'failure-screenshot.png' });
} finally {
  // always runs — for cleanup
  console.log('Test execution completed');
}

class TestAssertionError extends Error {

class APIError extends Error {

Read This First — What is MCP and Why Should QA Engineers Care?
As a QA engineer, you already use VS Code to write tests, Playwright to run them, and GitHub Copilot to help you code. But normally, Copilot can only suggest code — it cannot actually open a browser, click through your app, or see what happens on screen.

MCP changes that. MCP (Model Context Protocol) is a way to give AI tools like GitHub Copilot the ability to connect to and control real external tools — including a real browser using Playwright.

With Playwright MCP set up in VS Code, you can tell GitHub Copilot: "Open our login page, enter wrong credentials, and tell me what error message appears." Copilot will actually do it — open Chrome, go to the page, fill the fields, click the button, and report back what it saw. No manual testing needed from you.

This section teaches you exactly what MCP is and how to set it up in VS Code step by step.

What is MCP? — Model Context Protocol Explained Simply Start Here ⌄
🔗
Simple Explanation
Think of MCP like a USB port for AI. Your laptop has a USB port — you can plug in a keyboard, a mouse, a hard drive, or a camera. The laptop does not need to know in advance what you will plug in. It just has a standard port that anything can connect to.

MCP is the same idea — but for AI assistants. It is a standard way for AI tools like Claude to connect to external services — your file system, your browser, your Jira board, your database — without needing a custom integration for each one.

Technical Explanation
MCP (Model Context Protocol) is an open standard developed by Anthropic. It uses a client-server architecture:

— MCP Host: The AI application (Claude Desktop, Claude.ai) that uses external tools
— MCP Client: Lives inside the host — manages connections to MCP servers
— MCP Server: A programme that exposes capabilities to the AI

Three things MCP servers can provide:
— Tools: Actions the AI can take — "run this test", "create a Jira ticket"
— Resources: Data the AI can read — "read this file", "get this API response"
— Prompts: Pre-built prompt templates for common tasks

Real MCP examples for QA:
— Filesystem MCP: Claude reads your test files directly
— GitHub MCP: Claude checks pull requests and writes test cases for new features
— Jira MCP: Claude reads tickets and generates test plans
— Playwright MCP: Claude controls a real browser and runs tests
— Database MCP: Claude queries your database to verify test results
MCP is open source — any tool can build an MCP server. A growing list already supports it: GitHub, Jira, Slack, Google Drive, PostgreSQL, and more. See the full list at github.com/modelcontextprotocol/servers

Troubleshooting Common Issues
Playwright not showing in tools list:
— Check that .vscode/mcp.json is in the root of your project folder
— Reload VS Code (Ctrl+Shift+P → "Reload Window")
— Make sure Node.js is installed: node --version in terminal

Copilot not using browser:
— Make sure you are in Agent mode, not Ask mode
— Check that the Playwright tool is toggled ON in the Tools panel

Browser opens but test fails:
— The selector might not match your app — ask Copilot to inspect the page first
— Try: "Inspect the login page and find the correct selector for the email field"

npx command not found:
— Node.js is not installed or not in PATH
— Install from nodejs.org and restart VS Code
The key difference between VS Code and Claude Desktop setup: VS Code uses .vscode/mcp.json in your project folder. Claude Desktop uses claude_desktop_config.json in your system config folder. Both use the same @playwright/mcp package.
