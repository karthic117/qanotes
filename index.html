1
What is Manual Testing? Foundation ⌄
👀
Simple Explanation
Imagine you bought a new phone. Before using it daily, you try every app, check the camera, test the buttons, and make sure the charger works. You are doing manual testing! In software, a tester does the same thing — opens the app, tries every feature, and notes down anything that does not work as expected.
Technical Explanation
Manual testing is the process of executing test cases by a human tester without the use of automation tools. The tester interacts with the application under test (AUT) directly — entering inputs, observing outputs, and comparing actual results against expected results defined in test cases. It is most effective for exploratory testing, usability testing, and ad-hoc testing where human judgement is essential.
Manual testing is the starting point for every QA engineer. Even if you move to automation later, you must first understand how to test manually — because automation only does what you tell it to do.
2
Types of Manual Testing — Explained in Detail Core Concepts ⌄
🗂️
Simple Explanation
Testing is not just one activity — it is a collection of different approaches, each designed to catch a specific type of problem. A good QA tester knows which type to apply and when.
⚙️ 1. Functional Testing
Verifies that each feature of the application works exactly as specified in the requirements. It is the most fundamental type of testing — every feature must be tested before anything else.
What to check: Input fields accept correct values, buttons perform the right actions, forms submit data correctly, error messages appear when required.

Example: On a login page — does entering the correct email and password navigate the user to the dashboard? Does clicking "Forgot Password" open the reset flow?

When to use: After every new feature is developed, before release, and after any bug fix.
💨 2. Smoke Testing
A quick, high-level test to check whether the most critical functionality of the application is working. It is done before detailed testing begins — to confirm the build is stable enough to test further.
What to check: Can the app be launched? Can the user log in? Does the home page load? Are the main navigation links working?

Example: A new build is given by the developer. Before running 200 test cases, the tester first checks — does the app open? Does login work? If these fail, the build is rejected immediately without wasting time on further tests.

When to use: Every time a new build is received from the development team.
Smoke testing saves time. If the app cannot even open, there is no point testing anything else.
🎯 3. Sanity Testing
A focused check done after a bug fix or a small change — to confirm that the specific fix works and has not caused new issues in that area. Unlike smoke testing (which is broad), sanity testing is narrow and targeted.
What to check: Only the area that was changed or fixed is tested in detail.

Example: Developer fixes a bug where the "Apply Coupon" button was not working. The tester performs a sanity test — tests only the coupon functionality to confirm it now works correctly, without re-testing the entire checkout flow.

When to use: After a bug fix or a minor change, when there is limited time.
🔁 4. Regression Testing
Re-tests previously working functionality to ensure that new code changes have not broken anything. Every time the codebase changes, regression testing is done to protect existing features.
What to check: All features that were working before the change must still work after the change.

Example: A new "Dark Mode" feature is added to the app. Regression testing ensures that login, search, checkout, and all other existing features still work correctly — even though only Dark Mode was changed.

When to use: After every code change, bug fix, or new feature addition.
Regression testing is the most repeated type of testing in any project — this is why automation is often applied here first.
🔍 5. Exploratory Testing
The tester freely explores the application without predefined test cases — using knowledge, intuition, and curiosity to discover unexpected bugs. This relies entirely on the tester's experience and thinking.
What to check: Anything that feels unusual, unexpected, or poorly handled — edge cases that no one thought to write a test case for.

Example: The tester opens a shopping app and starts trying unusual things — adding 999 items to the cart, typing emojis in the search bar, rapidly clicking the checkout button multiple times, switching tabs mid-payment. These are scenarios developers may not have considered.

When to use: When exploring a new feature for the first time, or when looking for bugs that structured test cases might miss.
📏 6. Boundary Value Testing
Tests the extreme edges of allowed input ranges. Most defects occur at boundaries — not in the middle. This technique ensures the application handles minimum, maximum, and just-outside values correctly.
What to check: Minimum value, maximum value, one below minimum, one above maximum.

Example: A password field requires 8 to 20 characters.
— 7 characters → should be rejected
— 8 characters → should be accepted (minimum boundary)
— 20 characters → should be accepted (maximum boundary)
— 21 characters → should be rejected

When to use: Any field with a defined minimum or maximum — age, price, character count, quantity, date range.
❌ 7. Negative Testing
Tests the application with invalid, unexpected, or incorrect inputs — to verify it handles errors gracefully and does not crash or behave unpredictably. A well-built application should always respond with a clear, helpful error message.
What to check: Invalid data, empty required fields, wrong formats, special characters in unexpected places, SQL injection attempts.

Example:
— Enter letters in a numeric phone field → should show "Please enter a valid number"
— Submit a form with all fields empty → each required field should show a validation error
— Enter a past date in a "Date of Birth" field that requires adults only → should be rejected

When to use: For every input field and every user action — always test what happens when something goes wrong.
Negative testing is often skipped by beginners — but it is where many critical bugs hide. Always test the wrong path, not just the right one.
🖥️ 8. UI Testing
Verifies the visual and interactive elements of the application — layout, fonts, colours, button positions, spacing, and responsiveness across different screen sizes and browsers.
What to check: Alignment, font consistency, button sizes, colour contrast, broken images, overlapping elements, and how the page looks on mobile vs desktop.

Example:
— The Submit button is cut off on smaller phone screens
— A label overlaps the input field on Firefox but looks fine on Chrome
— An error message appears in white text on a white background — invisible to the user

When to use: After UI changes, after responsive design updates, and when testing on multiple browsers or devices.
👤 9. User Acceptance Testing (UAT)
The final stage of testing — done by the actual end users or business stakeholders to confirm the software meets their needs and is ready for release. UAT is not done by the QA team — it is done by the people who will actually use the product.
What to check: Whether the software solves the real problem it was built for, from the perspective of the actual user — not just technical correctness.

Example: A hospital management system is built and tested by the QA team. Before going live, actual hospital staff — doctors, nurses, front desk — are given access to try it in a test environment. They verify that the workflows match how they actually work in real life.

When to use: After all QA testing is complete, just before the official release.
UAT can catch issues that QA missed — not because QA was wrong, but because real users think and behave differently from testers.
⚡ 10. Ad-hoc Testing
Informal testing with no documentation, no test cases, and no plan — the tester randomly tests the application trying to break it. It relies entirely on the tester's intuition and domain knowledge.
What to check: Anything — there are no rules. The tester goes wherever their instinct leads.

Example: A tester opens an e-commerce app and starts randomly — searches for a product with only spaces, adds it to the cart, removes it, adds it again, applies a coupon, removes the coupon, goes back, refreshes — just to see if anything breaks unexpectedly.

When to use: When time is short, when exploring a completely new area of the app, or as a complement to structured testing.

How to Write a Test Case Must Know ⌄
📝
Simple Explanation
A test case is like a recipe. A recipe tells you: what ingredients you need, the steps to follow, and what the dish should look like when done. A test case tells you: what conditions are needed, what steps to perform, and what the correct result should be. Anyone should be able to follow a well-written test case and get the same result.
Technical Explanation
A test case is a documented set of conditions, inputs, steps, and expected outcomes used to validate a specific functionality. A well-written test case includes: Test Case ID, Title, Preconditions, Test Steps, Test Data, Expected Result, Actual Result, Status (Pass/Fail), and Priority (Critical/High/Medium/Low).

Example — Login Feature Test Case:

Test Case Format 
Test Case ID   : TC_LOGIN_001
Title          : Verify successful login with valid credentials
Priority       : Critical

Precondition   : User account must exist. App must be open.

Test Steps     :
  1. Open the application and navigate to the Login page
  2. Enter a valid email address in the Email field
  3. Enter the correct password in the Password field
  4. Click the Login button

Test Data      :
  Email    : testuser@example.com
  Password : Test@1234

Expected Result: User is navigated to the Dashboard page.
                  Welcome message is displayed with the user's name.

Actual Result  : [Fill after executing the test]
Status         : Pass / Fail
Always write test cases before you start testing — not while testing. This keeps your approach structured and ensures nothing is missed.

How to Write a Bug Report Must Know ⌄
🐛
Simple Explanation
When you find a problem, you need to report it clearly so the developer can fix it. A good bug report answers three questions: What did you do? What did you expect to happen? What actually happened? The clearer your report, the faster the developer finds and fixes the problem.
Technical Explanation
A defect report (bug report) is a formal document submitted to a bug tracking tool (Jira, Azure DevOps, Bugzilla) that captures a software defect. Severity indicates the impact on the system (Critical, High, Medium, Low). Priority indicates how urgently it must be fixed. Both are determined by the QA engineer and reviewed by the project manager.
Bug Report Format 
Bug ID         : BUG_LOGIN_003
Title          : Login button unresponsive when password field is empty
Severity       : High
Priority       : High
Environment    : Chrome 124, Windows 11, App v2.3.1

Steps to Reproduce:
  1. Open the application Login page
  2. Enter a valid email address
  3. Leave the Password field completely empty
  4. Click the Login button

Expected Result:
  A validation message should appear:
  "Password is required. Please enter your password."

Actual Result  :
  Nothing happens. The Login button does not respond.
  No error message is shown. No page navigation occurs.

Attachments    : Screenshot_login_empty_pass.png
Always attach a screenshot or screen recording to your bug report. It removes all confusion and helps the developer reproduce the issue immediately.

Test Life Cycle — How Testing Works in a Project Process ⌄
🔄
Simple Explanation
Testing in a real project follows a clear process — just like how a book goes from author → editor → proofreader → printer → bookshop. Each step has a purpose. Skipping a step causes problems later. This process is called the Software Testing Life Cycle (STLC).
  • Requirement Analysis — Read and understand what the feature should do. Ask questions if anything is unclear.
  • Test Planning — Decide what to test, how to test it, who will test it, and how long it will take.
  • Test Case Design — Write all test cases covering happy path, negative, boundary, and edge cases.
  • Test Environment Setup — Prepare the system, test data, and tools needed to run the tests.
  • Test Execution — Run each test case one by one. Mark the result as Pass or Fail.
  • Bug Reporting — Log every failure as a bug in the tracking tool with clear steps and screenshots.
  • Re-testing & Regression — After the developer fixes the bug, test it again. Also check that the fix did not break anything else.
  • Test Closure — Summarise the results, prepare a test summary report, and confirm the feature is ready to release.
Understanding this life cycle is the single most important foundation for any QA career. Master this first — everything else builds on top of it.

Manual vs Automation Testing — When to Use Which Key Decision ⌄
⚖️
Simple Explanation
A calculator is faster than doing arithmetic by hand — but it cannot decide which calculation to perform. Manual testing is like the human deciding what to check. Automation is like the calculator doing it quickly once you tell it what to do. Both are needed — they work best together.
Manual TestingAutomation Testing
Best forExploratory, usability, ad-hoc, one-time testsRepetitive, regression, data-driven tests
SpeedSlower — human executes each stepVery fast — runs hundreds of tests in minutes
CostLow initial cost, higher long-term effortHigher initial setup, lower long-term effort
Human judgement✅ Required — great for UI/UX feedback❌ Cannot judge "does this look right?"
ConsistencyCan vary — humans get tired✅ Always consistent — same steps every run
New features✅ Best — explore before automating❌ Not ideal until feature is stable
Learn first✅ Yes — start hereAfter manual skills are solid
Rule of thumb: Test manually first to understand the feature. Once a test is stable and needs to run repeatedly, automate it with Playwright.
Example — Read This First
Imagine you work at an e-commerce company. Every time your team adds a new feature — like a new checkout flow or a payment option — you need to test it on 10 different browsers and 5 different screen sizes. That is 50 tests. Doing it manually every day takes hours and delays releases.

Playwright runs all 50 tests automatically in under 2 minutes.

What is Playwright? Understanding ⌄
📺
Simple Explanation — Anyone Can Understand
You know how a TV remote controls the television without you going up to it and pressing buttons? Playwright is like a remote control for websites. It can open a browser, click buttons, fill forms, and tell you if something went wrong — all on its own, without any human clicking!
Technical Explanation — For Engineering Students
Playwright is an open-source end-to-end testing framework developed by Microsoft. It automates Chromium, Firefox, and WebKit browsers through a single unified API. It supports TypeScript and JavaScript. Key features include parallel test execution, network request interception, mobile device emulation, and built-in trace/video reporting for debugging.
Playwright is open-source and is trusted by companies like Microsoft and Google.

Installation — VS Code + Playwright Setup Setup ⌄
🖥️
Simple Explanation
We use Visual Studio Code (VS Code) as our editor. It is the most widely used code editor in the industry. Once VS Code is set up, we install the Playwright extension inside it — which lets you run, debug, and even record tests by clicking through the browser, without writing a single line of code manually.
Technical Explanation
The official VS Code extension for Playwright (ms-playwright.playwright) integrates with VS Code's Test Explorer panel. It enables one-click test execution, inline pass/fail indicators, step-through debugging with breakpoints, and Codegen — a test recorder that watches your browser actions and generates TypeScript test code automatically.

Step 1 — Install VS Code

  • Go to code.visualstudio.com and download VS Code for your operating system (Windows / Mac / Linux)
  • Run the installer and open VS Code once installation is complete

Step 2 — Install Node.js

  • Go to nodejs.org and download the LTS version (Long Term Support — the stable one)
  • Run the installer with default settings
  • Open VS Code → press Ctrl + ` (backtick) to open the built-in terminal → verify with:
VS Code Terminal 
node --version     # should show v18.0.0 or higher
npm --version      # should show 9 or higher

Step 3 — Install the Playwright Extension in VS Code

  • In VS Code, click the Extensions icon on the left sidebar (four squares icon)
  • Search for Playwright Test for VSCode
  • Click Install on the extension by Microsoft

Step 4 — Create a New Playwright Project

  • In VS Code, open the Command Palette with Ctrl + Shift + P
  • Type Install Playwright and select it from the list
  • A setup panel opens — choose these options:
Setup Options 
Language      → TypeScript
Test folder   → tests
GitHub CI     → Yes
Browsers      → Chromium, Firefox, WebKit (tick all three)
  • Click OK — VS Code will download all browser binaries automatically

Step 5 — Run Your First Test from VS Code

  • Click the Testing icon on the left sidebar (looks like a flask/beaker)
  • You will see the example tests that were created during setup
  • Click the ▶ Run All Tests button at the top
  • Tests run — green tick means pass, red cross means fail
  • To open the detailed HTML report, run this in the VS Code terminal:
VS Code Terminal 
npx playwright show-report   # opens report in browser

Bonus — Record a Test Without Writing Code

  • Open Command Palette → Ctrl + Shift + P
  • Type Record New and select Record new test
  • A browser opens — click through your app as a normal user
  • VS Code automatically writes the TypeScript test code as you click
  • Stop the recording — the test file is saved and ready to run
The Record feature is very useful for beginners — you interact with the app normally and Playwright generates the test code for you. You can then review and refine it.

Writing Your First Test — Login Page Practice ⌄
🏫
Simple Explanation
Think of a security checkpoint. It checks — is this person authorised to enter? A login test does the same — it verifies the right user gets in with valid credentials, and anyone with wrong credentials is blocked with a clear error message.
Example — What We Are Testing
A banking app has a login screen. We want to make sure:
✅ Correct email + password → goes to the account home page
❌ Wrong password entered → shows "Incorrect credentials"
❌ Empty password field → shows "Please enter your password"
Technical Explanation
test.describe() groups related tests together. page.fill() types text into an input field. page.click() presses a button. expect(page).toHaveURL() verifies that the browser navigated to the correct page. Each test runs in a completely fresh browser tab — no shared data between tests.
TypeScript 
// File: tests/login.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Bank App — Login Page', () => {

  // ✅ Test 1: Correct login — should go to home page
  test('valid user can log in successfully', async ({ page }) => {
    await page.goto('https://mybank.com/login');

    // Type the mobile number
    await page.fill('#mobile', '9876543210');

    // Type the PIN
    await page.fill('#pin', '1234');

    // Click the Login button
    await page.click('button[type="submit"]');

    // Verify: Did we reach the home page?
    await expect(page).toHaveURL('/home');
    await expect(page.locator('h1')).toContainText('Welcome');
  });

  // ❌ Test 2: Wrong PIN — should show an error message
  test('wrong PIN shows error message', async ({ page }) => {
    await page.goto('https://mybank.com/login');
    await page.fill('#mobile', '9876543210');
    await page.fill('#pin', '9999');  // wrong PIN
    await page.click('button[type="submit"]');

    // The error message must appear on screen
    await expect(page.locator('.error-message')).toBeVisible();
    await expect(page.locator('.error-message'))
      .toContainText('Incorrect PIN');
  });

});
Notice we tested both the good path (correct login) AND the bad path (wrong PIN). Always test both — that is what makes a thorough QA tester!

Page Object Model — Keeping Code Organised Good Practice ⌄
🗂️
Simple Explanation
Think of organising files on a computer. You keep documents in one folder, images in another, and code in another. Page Object Model does the same for test code — each page of the application gets its own separate file. Everything is organised and easy to find.
Example — Why This Matters
Your team has 30 test files all using page.fill('#password', ...). Tomorrow the developer renames that field to #user-password. Without POM, you update all 30 files — that takes half a day. With POM, you update just one file — done in a minute.
TypeScript 
// File: pages/LoginPage.ts  ← One neat file for the login page
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  readonly mobileInput: Locator;
  readonly pinInput: Locator;
  readonly loginButton: Locator;
  readonly errorMessage: Locator;

  constructor(private page: Page) {
    // Define where each element is located on the page
    this.mobileInput  = page.locator('#mobile');
    this.pinInput      = page.locator('#pin');
    this.loginButton   = page.locator('button[type="submit"]');
    this.errorMessage  = page.locator('.error-message');
  }

  // One helper action that performs the full login
  async login(mobile: string, pin: string) {
    await this.page.goto('/login');
    await this.mobileInput.fill(mobile);
    await this.pinInput.fill(pin);
    await this.loginButton.click();
  }
}

// ─────────────────────────────────────────────
// File: tests/login.spec.ts  ← Now much cleaner!
import { LoginPage } from '../pages/LoginPage';

test('manager login test', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.login('9000012345', '5678');
  await expect(page).toHaveURL('/manager-home');
});

Automated Testing in CI/CD — Tests That Run Themselves DevOps ⌄
🎯
Simple Explanation
Think of a quality check that runs automatically every time something changes on an assembly line. No one has to press a button — it just happens. CI/CD works the same way. Every time a developer saves new code, all tests run automatically. If something breaks, the team is notified immediately.
Example
A developer adds a new feature at 3 PM. The moment the code is saved to GitHub, 150 tests run automatically on a cloud computer. By 3:07 PM the team gets a notification: "✅ All 150 tests passed — safe to release." Or "❌ 3 tests failed — do not release yet." No manual work needed.
GitHub Actions — YAML Config File 
# File: .github/workflows/playwright.yml
# This tells GitHub: run our tests every time code is saved

name: Playwright Test Suite
on:
  push:
    branches: [main]       # run when code is pushed to main
  pull_request:
    branches: [main]       # run when someone wants to merge code

jobs:
  test:
    runs-on: ubuntu-latest  # use a cloud computer
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20 }
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: test-report
          path: playwright-report/  # save the report for team to review
GitHub Actions works with public repositories and is a good way for students to practise CI/CD with their own projects.
Example — Read This First
Imagine you need to prepare 100 test questions for a subject. That would take days. But if you tell an AI: "Write 100 multiple choice questions about networking with 4 options each" — it produces all 100 in under a minute.

QA testers use Gen AI the same way. Describe the feature, ask for test scenarios — the AI drafts them in seconds. You review, refine, and use them.

What is Generative AI? Understanding ⌄
📖
Simple Explanation
Imagine a student who has read every textbook, every newspaper, and every website ever written. Now ask that student any question — they give a detailed, well-structured answer because they have read so much! Generative AI is like that extraordinary student, but running on a computer. It can answer questions, write code, create reports, and much more.
Technical Explanation
Generative AI models (like GPT-4, Claude, Gemini) are Large Language Models (LLMs) trained on massive text datasets using transformer architecture. They predict the most likely next word/token based on the input context. For QA use cases, they understand natural language feature descriptions and produce structured outputs — test cases, scripts, defect reports — when given well-written prompts.
⚠️ Important: AI can sometimes give confident but incorrect answers. Always review and verify AI-generated test cases before using them in your work.

Generating Test Cases Using AI Core Skill ⌄
✍️
Simple Explanation
A good teacher prepares different types of questions before an exam — easy ones, tricky ones, and "what if" ones. You can ask an AI to do the same for your app's features. Just describe the feature clearly and say "what can go wrong?" — the AI will suggest many test scenarios you may not have thought of!
Example — Payment Feature Testing
You are testing the payment feature on an e-commerce app. You need to think of many ways it could fail — invalid card, insufficient balance, server timeout, duplicate payment, and more. Instead of typing all these manually, you describe the feature to AI and ask it to generate all scenarios at once.
AI Prompt — Copy and Use This 
You are a senior QA engineer with 10 years of experience.

I am testing a payment feature on an e-commerce app.
Here are the rules of this feature:
- User must be logged in to make a payment
- Minimum payment amount: $1
- Maximum payment amount: $10,000 per transaction
- Accepted: Credit card or Debit card
- Payment fails if card number is invalid
- Transaction times out after 3 minutes if not completed

Please generate test cases in this exact table format:
Test ID | Test Title | Steps to Follow | Expected Result | Priority

Include these types of tests:
✅ 3 successful payment tests (happy path)
❌ 5 failure tests (invalid card, wrong CVV, timeout, etc.)
🔲 3 boundary tests (minimum $1, maximum $10,000, just above limit)
⚡ 2 edge case tests (double tap on Pay button, network drops mid-payment)
The more clearly you describe the feature rules to the AI, the better and more accurate the test cases will be. Spend 2 minutes writing a good prompt to save 2 hours of manual work!

What is GitHub Copilot and How to Set It Up Getting Started ⌄
✈️

How to install in VS Code:

  • Open VS Code → Click the Extensions icon on the left sidebar
  • Search for GitHub Copilot → Click Install
  • Also search and install GitHub Copilot Chat (for asking questions)
  • Sign in with your GitHub account when prompted
  • Open any .ts test file and start typing — Copilot suggestions appear in faded grey text
  • Press Tab to accept a suggestion, or Esc to ignore it

Copilot Writing a Full Test From Your Comment See It in Action ⌄
💬
Simple Explanation
You describe what you want in a simple comment. Copilot reads your description and writes the full test code. You only type one line — Copilot types the next 20! It is like telling your helpful classmate "write this down for me" — and they do it neatly and correctly.
What You Type vs What Copilot Writes
You type only the comment on the first line. Copilot automatically suggests all the code below it:
TypeScript — Copilot Demo 
// YOU TYPE THIS ONE COMMENT ↓
// Test: student adds a course to cart on an ed-tech app, 
// goes to checkout, and verifies the course name and price appear

// COPILOT WRITES ALL OF THIS AUTOMATICALLY ↓
test('add course to cart and verify checkout', async ({ page }) => {
  await page.goto('/courses');

  // Get the name of the first course on the page
  const courseName = await page
    .locator('.course-card:first-child .course-title')
    .textContent();

  // Click Enroll Now
  await page
    .locator('.course-card:first-child .enroll-btn')
    .click();

  // Go to cart and verify the course appears
  await page.click('[data-testid="cart-icon"]');
  await expect(page.locator('.cart-item-title'))
    .toContainText(courseName!);

  // Proceed to checkout
  await page.click('button:has-text("Proceed to Pay")');
  await expect(page).toHaveURL('/checkout');
});
Write a detailed comment with the page name, user action, and what to verify. The more specific your comment, the better Copilot's suggestion!

Creating a Test Plan with Claude Planning ⌄
🗺️
Simple Explanation
Before starting any project, a good team prepares a plan — what will be done, by whom, in what order, and what counts as complete. A test plan works the same way. Before testing begins, we plan what to test, how to test it, what risks exist, and when testing is considered done.
Example — Test Plan
Your team is adding a "Express Checkout" feature to a booking app. You need a complete test plan before testing begins. Just describe it to Claude:
Claude Prompt — Test Plan 
You are a QA lead with 10 years of experience.

Create a complete test plan for this new feature:
"A flight booking app is adding an Express Checkout feature.
- Available only for registered users
- Saves card details for one-click payment
- Booking must be completed within 10 minutes of seat selection
- Price is locked at the time of seat selection
- Confirmation email is sent within 2 minutes of payment"

Please include:
1. Scope — what we will test and what we will NOT test
2. Types of testing needed (functional, performance, security, etc.)
3. Entry criteria — when testing can START
4. Exit criteria — when testing is considered COMPLETE
5. Risk areas — what could go wrong and cause problems
6. Test case summary table (Successful booking, Failed booking, Edge cases)
Claude can read very long documents — you can paste your entire requirements PDF text and it will read everything before creating the plan!

Writing a Professional Bug Report with Claude Bug Reporting ⌄
📝
Simple Explanation
If you call your internet provider and say "my internet is broken!" they cannot help you easily. But if you say "My internet disconnects every evening between 7 PM and 9 PM, only on my laptop, not on my phone, and the router light goes orange" — they can find the problem quickly!

Claude helps you write bug reports that are so clear and complete, any developer can immediately understand and fix the issue.
Claude Prompt — Bug Report 
Please format this as a professional bug report with these sections:
Bug ID | Summary | Steps to Reproduce | Expected Result | 
Actual Result | Severity (Critical / High / Medium / Low) |
Affected Environment | Suggested Root Cause | Screenshots Needed

Bug description (in my own words):
"When a user tries to pay on the checkout page 
and the cart has more than 5 items, the Pay Now button 
stops responding. The page freezes for about 30 seconds 
and then shows a blank white screen. The payment is NOT 
processed. This issue happens only on Android phones using 
Chrome browser. It works fine on desktop and iPhone."
After Claude formats the bug report, copy it directly into Jira, Bugzilla, or your team's tracking tool. This saves 15 to 20 minutes per bug report!
