What is Manual Testing? Foundation ⌄
👀
Simple Explanation
Imagine you bought a new phone. Before using it daily, you try every app, check the camera, test the buttons, and make sure the charger works. You are doing manual testing! In software, a tester does the same thing — opens the app, tries every feature, and notes down anything that does not work as expected.
Technical Explanation
Manual testing is the process of executing test cases by a human tester without the use of automation tools. The tester interacts with the application under test (AUT) directly — entering inputs, observing outputs, and comparing actual results against expected results defined in test cases. It is most effective for exploratory testing, usability testing, and ad-hoc testing where human judgement is essential.
Manual testing is the starting point for every QA engineer. Even if you move to automation later, you must first understand how to test manually — because automation only does what you tell it to do.

Types of Manual Testing — Explained in Detail Core Concepts ⌄
🗂️
Simple Explanation
Testing is not just one activity — it is a collection of different approaches, each designed to catch a specific type of problem. A good QA tester knows which type to apply and when.
⚙️ 1. Functional Testing
Verifies that each feature of the application works exactly as specified in the requirements. It is the most fundamental type of testing — every feature must be tested before anything else.
What to check: Input fields accept correct values, buttons perform the right actions, forms submit data correctly, error messages appear when required.

Example: On a login page — does entering the correct email and password navigate the user to the dashboard? Does clicking "Forgot Password" open the reset flow?

When to use: After every new feature is developed, before release, and after any bug fix.
💨 2. Smoke Testing
A quick, high-level test to check whether the most critical functionality of the application is working. It is done before detailed testing begins — to confirm the build is stable enough to test further.
What to check: Can the app be launched? Can the user log in? Does the home page load? Are the main navigation links working?

Example: A new build is given by the developer. Before running 200 test cases, the tester first checks — does the app open? Does login work? If these fail, the build is rejected immediately without wasting time on further tests.

When to use: Every time a new build is received from the development team.
Smoke testing saves time. If the app cannot even open, there is no point testing anything else.
🎯 3. Sanity Testing
A focused check done after a bug fix or a small change — to confirm that the specific fix works and has not caused new issues in that area. Unlike smoke testing (which is broad), sanity testing is narrow and targeted.
What to check: Only the area that was changed or fixed is tested in detail.

Example: Developer fixes a bug where the "Apply Coupon" button was not working. The tester performs a sanity test — tests only the coupon functionality to confirm it now works correctly, without re-testing the entire checkout flow.

When to use: After a bug fix or a minor change, when there is limited time.
🔁 4. Regression Testing
Re-tests previously working functionality to ensure that new code changes have not broken anything. Every time the codebase changes, regression testing is done to protect existing features.
What to check: All features that were working before the change must still work after the change.

Example: A new "Dark Mode" feature is added to the app. Regression testing ensures that login, search, checkout, and all other existing features still work correctly — even though only Dark Mode was changed.

When to use: After every code change, bug fix, or new feature addition.
Regression testing is the most repeated type of testing in any project — this is why automation is often applied here first.
🔍 5. Exploratory Testing
The tester freely explores the application without predefined test cases — using knowledge, intuition, and curiosity to discover unexpected bugs. This relies entirely on the tester's experience and thinking.
What to check: Anything that feels unusual, unexpected, or poorly handled — edge cases that no one thought to write a test case for.

Example: The tester opens a shopping app and starts trying unusual things — adding 999 items to the cart, typing emojis in the search bar, rapidly clicking the checkout button multiple times, switching tabs mid-payment. These are scenarios developers may not have considered.

When to use: When exploring a new feature for the first time, or when looking for bugs that structured test cases might miss.
📏 6. Boundary Value Testing
Tests the extreme edges of allowed input ranges. Most defects occur at boundaries — not in the middle. This technique ensures the application handles minimum, maximum, and just-outside values correctly.
What to check: Minimum value, maximum value, one below minimum, one above maximum.

Example: A password field requires 8 to 20 characters.
— 7 characters → should be rejected
— 8 characters → should be accepted (minimum boundary)
— 20 characters → should be accepted (maximum boundary)
— 21 characters → should be rejected

When to use: Any field with a defined minimum or maximum — age, price, character count, quantity, date range.
❌ 7. Negative Testing
Tests the application with invalid, unexpected, or incorrect inputs — to verify it handles errors gracefully and does not crash or behave unpredictably. A well-built application should always respond with a clear, helpful error message.
What to check: Invalid data, empty required fields, wrong formats, special characters in unexpected places, SQL injection attempts.

Example:
— Enter letters in a numeric phone field → should show "Please enter a valid number"
— Submit a form with all fields empty → each required field should show a validation error
— Enter a past date in a "Date of Birth" field that requires adults only → should be rejected

When to use: For every input field and every user action — always test what happens when something goes wrong.
Negative testing is often skipped by beginners — but it is where many critical bugs hide. Always test the wrong path, not just the right one.
🖥️ 8. UI Testing
Verifies the visual and interactive elements of the application — layout, fonts, colours, button positions, spacing, and responsiveness across different screen sizes and browsers.
What to check: Alignment, font consistency, button sizes, colour contrast, broken images, overlapping elements, and how the page looks on mobile vs desktop.

Example:
— The Submit button is cut off on smaller phone screens
— A label overlaps the input field on Firefox but looks fine on Chrome
— An error message appears in white text on a white background — invisible to the user

When to use: After UI changes, after responsive design updates, and when testing on multiple browsers or devices.
👤 9. User Acceptance Testing (UAT)
The final stage of testing — done by the actual end users or business stakeholders to confirm the software meets their needs and is ready for release. UAT is not done by the QA team — it is done by the people who will actually use the product.
What to check: Whether the software solves the real problem it was built for, from the perspective of the actual user — not just technical correctness.

Example: A hospital management system is built and tested by the QA team. Before going live, actual hospital staff — doctors, nurses, front desk — are given access to try it in a test environment. They verify that the workflows match how they actually work in real life.

When to use: After all QA testing is complete, just before the official release.
UAT can catch issues that QA missed — not because QA was wrong, but because real users think and behave differently from testers.
⚡ 10. Ad-hoc Testing
Informal testing with no documentation, no test cases, and no plan — the tester randomly tests the application trying to break it. It relies entirely on the tester's intuition and domain knowledge.
What to check: Anything — there are no rules. The tester goes wherever their instinct leads.

Example: A tester opens an e-commerce app and starts randomly — searches for a product with only spaces, adds it to the cart, removes it, adds it again, applies a coupon, removes the coupon, goes back, refreshes — just to see if anything breaks unexpectedly.

When to use: When time is short, when exploring a completely new area of the app, or as a complement to structured testing.

How to Write a Test Case Must Know ⌄
📝
Simple Explanation
A test case is like a recipe. A recipe tells you: what ingredients you need, the steps to follow, and what the dish should look like when done. A test case tells you: what conditions are needed, what steps to perform, and what the correct result should be. Anyone should be able to follow a well-written test case and get the same result.
Technical Explanation
A test case is a documented set of conditions, inputs, steps, and expected outcomes used to validate a specific functionality. A well-written test case includes: Test Case ID, Title, Preconditions, Test Steps, Test Data, Expected Result, Actual Result, Status (Pass/Fail), and Priority (Critical/High/Medium/Low).

Example — Login Feature Test Case:

Test Case Format 
Test Case ID   : TC_LOGIN_001
Title          : Verify successful login with valid credentials
Priority       : Critical

Precondition   : User account must exist. App must be open.

Test Steps     :
  1. Open the application and navigate to the Login page
  2. Enter a valid email address in the Email field
  3. Enter the correct password in the Password field
  4. Click the Login button

Test Data      :
  Email    : testuser@example.com
  Password : Test@1234

Expected Result: User is navigated to the Dashboard page.
                  Welcome message is displayed with the user's name.

Actual Result  : [Fill after executing the test]
Status         : Pass / Fail
Always write test cases before you start testing — not while testing. This keeps your approach structured and ensures nothing is missed.

When you find a problem, you need to report it clearly so the developer can fix it. A good bug report answers three questions: What did you do? What did you expect to happen? What actually happened? The clearer your report, the faster the developer finds and fixes the problem.

A defect report (bug report) is a formal document submitted to a bug tracking tool (Jira, Azure DevOps, Bugzilla) that captures a software defect. Severity indicates the impact on the system (Critical, High, Medium, Low). Priority indicates how urgently it must be fixed. Process ⌄
🔄
Simple Explanation
Testing in a real project follows a clear process — just like how a book goes from author → editor → proofreader → printer → bookshop. Each step has a purpose. Skipping a step causes problems later. This process is called the Software Testing Life Cycle (STLC).
  • Requirement Analysis — Read and understand what the feature should do. Ask questions if anything is unclear.
  • Test Planning — Decide what to test, how to test it, who will test it, and how long it will take.
  • Test Case Design — Write all test cases covering happy path, negative, boundary, and edge cases.
  • Test Environment Setup — Prepare the system, test data, and tools needed to run the tests.
  • Test Execution — Run each test case one by one. Mark the result as Pass or Fail.
  • Bug Reporting — Log every failure as a bug in the tracking tool with clear steps and screenshots.
  • Re-testing & Regression — After the developer fixes the bug, test it again. Also check that the fix did not break anything else.
  • Test Closure — Summarise the results, prepare a test summary report, and confirm the feature is ready to release.
Understanding this life cycle is the single most important foundation for any QA career. Master this first — everything else builds on top of it.

Entry Criteria and Exit Criteria — When Testing Starts and Ends Process ⌄
🚦
Simple Explanation
Think of a cricket match. The match only starts when the pitch is ready, both teams have arrived, and the umpires are in place — these are Entry Criteria. The match ends only when all overs are bowled or one team wins — these are Exit Criteria. In testing, we define similar conditions to decide when testing can start and when it is considered complete.
Technical Explanation
Entry Criteria — Conditions that must be met before testing begins. Ensures testers are not wasting time on an unstable build.

Exit Criteria — Conditions that must be met before testing is declared complete. Ensures the product meets the required quality standard before release.
Entry and exit criteria are written in the Test Plan. They protect the team — entry criteria prevent testing too early, exit criteria prevent releasing too soon.

7 Principles of Software Testing Foundation ⌄
📐
Simple Explanation
Just like there are rules in a game, software testing has 7 fundamental principles every QA engineer must know. These principles guide how we think about testing — they help us test smarter, not just more. Knowing these principles is the difference between a beginner tester and a professional QA engineer.
1️⃣
Testing shows presence of bugs
Testing can prove bugs exist — but can never prove the software is 100% bug-free. Even after thorough testing, hidden bugs may remain.
2️⃣
Exhaustive testing is impossible
You cannot test every possible combination of inputs. Focus effort on high-risk and high-impact areas using risk-based testing.
3️⃣
Early testing saves money
A bug found during requirements costs almost nothing to fix. The same bug found after release can cost 100x more. Start testing early.
4️⃣
Defect clustering
Most bugs are found in a small number of modules. 80% of bugs come from 20% of the code. If one area is buggy — dig deeper.
5️⃣
Pesticide paradox
If you run the same tests repeatedly, they stop finding new bugs — like a pesticide that stops working. Keep updating your test cases.
6️⃣
Testing is context-dependent
How you test a banking app differs from testing a game. Testing approach depends on the type of software, risk, and user.
7️⃣
Absence of errors fallacy
A bug-free product is useless if it does not meet user needs. If you built the wrong thing perfectly, it is still a failure. QA must verify the software solves the right problem.
These 7 principles are asked in almost every QA interview. Learn them thoroughly — they are the foundation of professional software testing.

Simple Explanation
In the old way, the team spent 6 months building everything, then 2 months testing, then released. If something was wrong, it was too late to fix easily.

Agile is different. The team works in short cycles called Sprints (usually 2 weeks). In each sprint, a small piece of the feature is built AND tested together. At the end of every sprint, the team has something working. Testing is not a phase at the end — it happens continuously throughout the project.

Agile is a software development methodology that delivers working software in short iterative cycles called Sprints (1–4 weeks). Key Agile ceremonies QA participates in:

— Sprint Planning — QA reviews user stories and raises questions before development starts
— Daily Standup — QA reports what was tested, what will be tested, any blockers
— Sprint Review — QA demonstrates tested features to stakeholders
— Retrospective — Team discusses what went well and what to improve next sprint

What is Playwright? — Full Definition Definition ⌄
📺
Simple Explanation
You know how a TV remote controls the television without you going up to it and pressing buttons? Playwright is like a remote control for websites. It can open a browser, click buttons, fill forms, scroll pages, take screenshots, and tell you if something went wrong — all on its own, without any human clicking.

When you write a Playwright test, you are basically writing a set of instructions that says: "Open Chrome, go to this website, click Login, type the username and password, click Submit, and check that the dashboard appears." Playwright follows those instructions perfectly every single time.
Technical Definition
Playwright is an open-source end-to-end (E2E) browser automation framework developed by Microsoft, released in 2020. It allows you to write test scripts in TypeScript, JavaScript, Python, Java, or C# that control real browsers programmatically.

Key capabilities:
— Automates Chromium (Chrome/Edge), Firefox, and WebKit (Safari) using one unified API
— Supports parallel execution — runs multiple tests at the same time to save time
— Built-in auto-wait — waits for elements to be ready before interacting, reducing flaky tests
— Network interception — can mock API responses for isolated testing
— Mobile emulation — simulates iPhone, Android devices
— Trace Viewer — records every test step with screenshots and network logs for debugging
— Codegen — records your browser actions and generates test code automatically
FeatureWhat It Means
End-to-End TestingTests the full user journey from start to finish — login → browse → checkout → confirmation
Cross-BrowserSame test runs on Chrome, Firefox, and Safari without rewriting
Headless ModeRuns tests without opening a visible browser window — faster for CI/CD pipelines
Auto-WaitAutomatically waits for buttons, forms, and pages to be ready — no need for manual sleep/wait calls
Parallel TestsRuns 10 tests at the same time instead of one-by-one — saves significant time
Playwright is the most popular automation tool in 2024-2025. It has replaced Selenium in many companies because it is faster, more reliable, and easier to write.

Writing Your First Test — Login Page Practice ⌄
🏫
Simple Explanation
Think of a security checkpoint. It checks — is this person authorised to enter? A login test does the same — it verifies the right user gets in with valid credentials, and anyone with wrong credentials is blocked with a clear error message.
Example — What We Are Testing
A banking app has a login screen. We want to make sure:
✅ Correct email + password → goes to the account home page
❌ Wrong password entered → shows "Incorrect credentials"
❌ Empty password field → shows "Please enter your password"
Technical Explanation
test.describe() groups related tests together. page.fill() types text into an input field. page.click() presses a button. expect(page).toHaveURL() verifies that the browser navigated to the correct page. Each test runs in a completely fresh browser tab — no shared data between tests.
TypeScript 
// File: tests/login.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Bank App — Login Page', () => {

  // ✅ Test 1: Correct login — should go to home page
  test('valid user can log in successfully', async ({ page }) => {
    await page.goto('https://mybank.com/login');

    // Type the mobile number
    await page.fill('#mobile', '9876543210');

    // Type the PIN
    await page.fill('#pin', '1234');

    // Click the Login button
    await page.click('button[type="submit"]');

    // Verify: Did we reach the home page?
    await expect(page).toHaveURL('/home');
    await expect(page.locator('h1')).toContainText('Welcome');
  });

  // ❌ Test 2: Wrong PIN — should show an error message
  test('wrong PIN shows error message', async ({ page }) => {
    await page.goto('https://mybank.com/login');
    await page.fill('#mobile', '9876543210');
    await page.fill('#pin', '9999');  // wrong PIN
    await page.click('button[type="submit"]');

    // The error message must appear on screen
    await expect(page.locator('.error-message')).toBeVisible();
    await expect(page.locator('.error-message'))
      .toContainText('Incorrect PIN');
  });

});
Notice we tested both the good path (correct login) AND the bad path (wrong PIN). Always test both — that is what makes a thorough QA tester!

Page Object Model — Keeping Code Organised Good Practice ⌄
🗂️
Simple Explanation
Think of organising files on a computer. You keep documents in one folder, images in another, and code in another. Page Object Model does the same for test code — each page of the application gets its own separate file. Everything is organised and easy to find.
Example — Why This Matters
Your team has 30 test files all using page.fill('#password', ...). Tomorrow the developer renames that field to #user-password. Without POM, you update all 30 files — that takes half a day. With POM, you update just one file — done in a minute.
TypeScript 
// File: pages/LoginPage.ts  ← One neat file for the login page
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  readonly mobileInput: Locator;
  readonly pinInput: Locator;
  readonly loginButton: Locator;
  readonly errorMessage: Locator;

  constructor(private page: Page) {
    // Define where each element is located on the page
    this.mobileInput  = page.locator('#mobile');
    this.pinInput      = page.locator('#pin');
    this.loginButton   = page.locator('button[type="submit"]');
    this.errorMessage  = page.locator('.error-message');
  }

  // One helper action that performs the full login
  async login(mobile: string, pin: string) {
    await this.page.goto('/login');
    await this.mobileInput.fill(mobile);
    await this.pinInput.fill(pin);
    await this.loginButton.click();
  }
}

// ─────────────────────────────────────────────
// File: tests/login.spec.ts  ← Now much cleaner!
import { LoginPage } from '../pages/LoginPage';

test('manager login test', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.login('9000012345', '5678');
  await expect(page).toHaveURL('/manager-home');
});

Automated Testing in CI/CD — Tests That Run Themselves DevOps ⌄

How Playwright Works — Step by Step with Diagram Concept ⌄
🔄
Simple Explanation
When you run a Playwright test, a lot happens behind the scenes in a specific order. Your code sends instructions → Playwright talks to the browser → the browser performs the action → Playwright checks the result → it reports Pass or Fail. Understanding this flow helps you write better tests and debug failures faster.
What Happens at Each Step
Step 1 — Your Test Code: You write instructions in TypeScript. Example: "go to login page, fill email, fill password, click submit, check dashboard appears."

Step 2 — Playwright Test Runner: Playwright reads your instructions and translates them into browser commands using the Chrome DevTools Protocol (CDP) or WebKit/Firefox equivalents.

Step 3 — Browser: A real browser (Chrome, Firefox, or Safari) opens and performs the actions — just like a human would, but much faster.

Step 4 — Website: The website responds to each action — form submissions, page navigations, API calls happen in real time.

Step 5 — Report: Playwright compares the result with what you expected. If they match → ✅ Pass. If not → ❌ Fail with a detailed error message and screenshot.
Playwright communicates with browsers using the same protocol that browser DevTools uses — so it can see everything a developer sees.

Locators — How Playwright Finds Elements on a Page Core Skill ⌄
🎯
Simple Explanation
Imagine you are in a large library and you need to find a specific book. You could find it by shelf number, by title, by author, or by the colour of its cover. Playwright finds elements on a webpage the same way — using different methods to identify the exact button, input field, or text it needs to interact with. These methods are called Locators.
TypeScript — All Locator Types 
// 1. By CSS ID — finds element with id="email"
await page.locator('#email').fill('test@example.com');

// 2. By Role — finds a button by its accessible role
await page.getByRole('button', { name: 'Login' }).click();

// 3. By Label — finds input associated with a label
await page.getByLabel('Email Address').fill('user@test.com');

// 4. By Placeholder — finds input with placeholder text
await page.getByPlaceholder('Enter your email').fill('user@test.com');

// 5. By Text — finds element containing specific text
await page.getByText('Forgot Password?').click();

// 6. By Test ID — uses data-testid attribute (most stable)
await page.getByTestId('submit-button').click();
Ask your developers to add data-testid attributes to important elements. This is the most stable locator — it does not break when the UI changes.

Assertions — How Playwright Checks Results Core Skill ⌄
✅
Simple Explanation
When a teacher marks an exam, they compare your answer with the correct answer. If they match — you pass. If not — you fail. Playwright does the same thing. After performing an action (like clicking Login), it checks the result (like "is the dashboard visible?"). These checks are called Assertions and they are what make a test actually useful.
TypeScript — Common Assertions 
// Check the page URL changed correctly
await expect(page).toHaveURL('/dashboard');

// Check page title
await expect(page).toHaveTitle('Welcome — My App');

// Check an element is visible
await expect(page.locator('.success-message')).toBeVisible();

// Check an element contains specific text
await expect(page.locator('h1')).toContainText('Welcome, Karthic');

// Check an input has a specific value
await expect(page.locator('#email')).toHaveValue('test@example.com');

// Check a button is disabled
await expect(page.getByRole('button', { name: 'Submit' })).toBeDisabled();

// Check element count
await expect(page.locator('.product-card')).toHaveCount(12);
Every test must have at least one assertion. A test without an assertion always passes — even if the app is broken. Always verify the result!

TypeScript — QA Variables 
// ── DECLARING VARIABLES WITH TYPES ──────────────────

// string  = text (always in quotes)
let testName: string  = 'Login Test';
let browserName: string = 'Chrome';
let testerName: string  = 'Karthic';

// number  = any number (whole or decimal)
let testCaseId: number  = 1001;
let totalTests: number  = 50;
let passedTests: number = 47;
let duration: number    = 1.5;   // seconds

// boolean = true or false ONLY
let isPassed: boolean    = true;
let isBlocked: boolean   = false;
let isRegression: boolean = true;

// ── TYPE SAFETY IN ACTION ────────────────────────────
// TypeScript catches this mistake BEFORE running:
testName = 123;   // ❌ Error: number is not assignable to type string
testCaseId = 'TC-001'; // ❌ Error: string is not assignable to type number

// ── const vs let ─────────────────────────────────────
// const = value NEVER changes (use for fixed config)
const baseUrl: string = 'https://myapp.com';

// let = value CAN change (use for test results)

7
Async / Await — How Playwright Tests Work Must Know ⌄
⏳
Simple Explanation
When you send a WhatsApp message, your phone does not freeze and wait until the message is delivered before you can do anything else. It sends the message in the background and continues. This is asynchronous behaviour.

Playwright tests are asynchronous because browsers take time to respond — a page might take 2 seconds to load. await tells your test: "wait here until this browser action is complete before moving to the next step." Without await, your test would run all steps simultaneously and fail unpredictably.
TypeScript — Async/Await in Playwright 
// Every Playwright test function must be async
// Every browser action must be awaited

import { test, expect } from '@playwright/test';

// ── BASIC ASYNC TEST ─────────────────────────────────
test('User can log in successfully', async ({ page }) => {

  // await = wait for this to finish before next step
  await page.goto('https://myapp.com/login');
  // ↑ waits for page to fully load

  await page.fill('#email', 'test@example.com');
  // ↑ waits for email field to accept text

  await page.fill('#password', 'Test@1234');
  // ↑ waits for password field to accept text

  await page.click('button[type="submit"]');
  // ↑ waits for button click to register

  await expect(page).toHaveURL('/dashboard');
  // ↑ waits for navigation to complete, then checks URL
});

// ── ASYNC HELPER FUNCTION ────────────────────────────
// async function = a function that contains await steps
async function loginUser(page: Page, email: string, password: string): Promise<void> {
  await page.goto('/login');
  await page.fill('#email', email);
  await page.fill('#password', password);
  await page.click('button[type="submit"]');
}

// Using the helper in multiple tests
test('Dashboard loads after login', async ({ page }) => {
  await loginUser(page, 'test@example.com', 'Test@1234');
  await expect(page.locator('h1')).toContainText('Welcome');
});
The most common beginner mistake in Playwright is forgetting await. If you forget it on a step, the test moves to the next line immediately — before the browser finishes the action — and the test fails for no obvious reason. Always add await before every browser action.

Classes — Building Page Object Models OOP ⌄
🏗️
Simple Explanation
A class is a template. Think of it like a cookie cutter — the cutter defines the shape, and every cookie made with it has the same shape but can have different fillings. In Playwright, we use classes to represent each page of the application (Login page, Dashboard page, Checkout page). Each class holds all the selectors and actions for that page in one organised place.
TypeScript — Page Object Model with Classes 
// ── BASE PAGE (shared actions for all pages) ─────────
class BasePage {
  protected page: Page;

  constructor(page: Page) {
    this.page = page;
  }

  async waitForPageLoad(): Promise<void> {
    await this.page.waitForLoadState('networkidle');
  }

  async getTitle(): Promise<string> {
    return await this.page.title();
  }
}

// ── LOGIN PAGE (extends BasePage) ────────────────────
class LoginPage extends BasePage {
  // Selectors — stored as class properties
  private emailInput    = '#email';
  private passwordInput = '#password';
  private loginButton   = 'button[type="submit"]';
  private errorMessage  = '.error-msg';

  // Go to login page
  async goto(): Promise<void> {
    await this.page.goto('/login');
  }

  // Login action — combines all steps
  async login(email: string, password: string): Promise<void> {
    await this.page.fill(this.emailInput, email);
    await this.page.fill(this.passwordInput, password);
    await this.page.click(this.loginButton);
  }

  // Get error message text
  async getError(): Promise<string> {
    return await this.page.locator(this.errorMessage).textContent() ?? '';
  }
}

// ── USING IN A TEST ──────────────────────────────────
test('Valid login redirects to dashboard', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('test@example.com', 'Test@1234');
  await expect(page).toHaveURL('/dashboard');
});

test('Wrong password shows error message', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('test@example.com', 'WrongPass');
  const error = await loginPage.getError();
  await expect(error).toContain('Invalid credentials');
});
Page Object Model (POM) using classes is the industry standard for organising Playwright tests. When a selector changes in the app, you update it in ONE place — the class file — and all tests that use it are automatically fixed.

Error Handling — Writing Reliable Tests Reliability ⌄
🛡️
Simple Explanation
When you drive a car, you wear a seatbelt — not because you plan to crash, but just in case. Error handling in TypeScript works the same way. You wrap your risky code in a try block. If anything goes wrong, the catch block handles it gracefully — instead of crashing the entire test run and giving you no useful information about what failed.
TypeScript — Error Handling in Tests 
// ── BASIC TRY-CATCH ──────────────────────────────────
try {
  // code that might fail goes here
  await page.goto('https://myapp.com/dashboard');
  await expect(page.locator('h1')).toBeVisible();
} catch (error) {
  // runs if anything above fails
  console.error('Test failed:', (error as Error).message);
  await page.screenshot({ path: 'failure-screenshot.png' });
} finally {
  // always runs — for cleanup
  console.log('Test execution completed');
}

// ── CUSTOM ERROR CLASS FOR QA ────────────────────────
class TestAssertionError extends Error {
  public testId: string;
  public severity: string;
  public screenshot: string;

  constructor(testId: string, message: string, severity: string) {
    super(message);
    this.name       = 'TestAssertionError';
    this.testId     = testId;
    this.severity   = severity;
    this.screenshot = `screenshots/${testId}_failure.png`;
  }
}

// Throwing and catching custom error
try {
  const url = await page.url();
  if (!url.includes('/dashboard')) {
    throw new TestAssertionError(
      'TC_001',
      `Expected /dashboard but got: ${url}`,
      'Critical'
    );
  }
} catch (e) {
  if (e instanceof TestAssertionError) {
    console.error(`[${e.severity}] ${e.testId}: ${e.message}`);
    console.error(`Screenshot: ${e.screenshot}`);
  }
}
Always add a screenshot in the catch block of failing tests. This is the most valuable debugging tool — you can see exactly what the browser showed at the moment of failure.

Generics — Writing Flexible, Reusable Code Advanced ⌄
🔧
Simple Explanation
Imagine a delivery box that can hold any type of item — a book, a phone, or food — but once you put something in, it only allows that same type. Generics in TypeScript work the same way. You write one function or interface that works with any type, and TypeScript keeps track of which type is being used so it can still catch mistakes.
TypeScript — Generics for QA

// <T> is a placeholder — replaced with real type when used
interface ApiResponse<T> {
  success: boolean;
  data: T;               // T = whatever type you specify
  message: string;
  statusCode: number;
  timestamp: string;
}

// User interface
interface User {
  id: number;
  name: string;
  email: string;
  role: string;
}

// Product interface
interface Product {
  id: number;
  name: string;
  price: number;
  inStock: boolean;
}

// ApiResponse<User> — data field is a User object
const userResp: ApiResponse<User> = {
  success: true,
  data: { id: 1, name: 'Karthic', email: 'k@test.com', role: 'QA' },

  timestamp: '2025-01-15T10:30:00Z'

};

// TypeScript knows the exact type of data in each response

console.log(prodResp.data[1].inStock);  // false

// ── GENERIC FUNCTION ─────────────────────────────────
function assertApiSuccess<T>(response: ApiResponse<T>): T {
  if (!response.success) {
    throw new Error(`API failed: ${response.message} (${response.statusCode})`);
  }
  return response.data;
}

const user    = assertApiSuccess(userResp);   // type: User
const products = assertApiSuccess(prodResp);  // type: Product[]

Type Guards — Safely Checking Types at Runtime Advanced ⌄
🔎
Simple Explanation
Sometimes you receive data from an API or a function and you are not 100% sure what type it is. Before using it, you want to check — "is this a string or a number?" A Type Guard is a condition that checks the type at runtime and narrows it down so TypeScript knows exactly what you are working with inside that block.
TypeScript — Type Guards in QA 
// ── typeof GUARD ─────────────────────────────────────
function printTestId(id: number | string): void {
  if (typeof id === 'number') {
    console.log(`Numeric ID: TC-${id.toString().padStart(4, '0')}`);
    // ↑ TypeScript knows id is a number here
  } else {
    console.log(`String ID: ${id.toUpperCase()}`);
    // ↑ TypeScript knows id is a string here
  }
}
printTestId(42);           // Numeric ID: TC-0042
printTestId('tc-login');   // String ID: TC-LOGIN

// ── instanceof GUARD ─────────────────────────────────
class APIError extends Error {
  constructor(public statusCode: number, message: string) {
    super(message);
    this.name = 'APIError';
  }
}

class TimeoutError extends Error {
  constructor(public timeoutMs: number, message: string) {
    super(message);
    this.name = 'TimeoutError';
  }
}

function handleTestError(error: APIError | TimeoutError | Error): void {
  if (error instanceof APIError) {
    console.error(`API Error ${error.statusCode}: ${error.message}`);
  } else if (error instanceof TimeoutError) {
    console.error(`Timeout after ${error.timeoutMs}ms: ${error.message}`);
  } else {
    console.error(`General Error: ${error.message}`);
  }
}

handleTestError(new TimeoutError(30000, 'Page did not load'));

This is how professional QA teams structure their Playwright projects. Types are defined once in the types/ folder, page actions are in the pages/ folder, and test files in tests/ are clean and readable — no messy selectors or repeated logic.
